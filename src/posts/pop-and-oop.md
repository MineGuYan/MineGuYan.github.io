## 面向过程编程（POP）与面向对象编程（OOP）

### 一、历史背景

#### 1. 面向过程编程

20世纪50年代至60年代，早期高级编程语言（如C语言）均采用面向过程编程作为主要编程范式，强调‌**函数、过程和顺序执行**‌，程序围绕“如何一步步解决问题”展开，即**做什么、按什么步骤做**。

但随着软件规模的扩大，面向过程编程的代码量迅速增加，变得十分**难以维护和扩展**。

#### 2. 面向对象编程

因此在七十年代，面向对象编程出现了。面向对象编程在面向过程编程的基础上引入‌了**封装、继承、多态**‌等机制，其核心目的是为了进一步封装代码，提高代码**可复用性和可维护性**。

### 二、面向对象编程三大机制

#### 1. 封装

将强关联的**数据**（也称属性）和操作数据的**方法**（也称行为）捆绑在一起，形成一个独立的单元（即对象），并将一部分信息隐藏起来，仅暴露出特定接口，与外部进行交互。即**捆绑**和**信息隐藏**。

函数和结构体其实就是一种封装，不过前者仅是行为的封装，后者仅是对数据的封装。而**类是对数据和方法两者的封装**。

**优点**：

- **安全性**：保护内部数据不被随意修改。
- **简化使用**：外部使用者无需了解复杂的内部实现，只需调用公开方法。
- **降低耦合**：只要公开接口不变，对象内部的修改不会影响使用它的其他部分，大大提高了代码的可维护性。

#### 2. 继承

是一种建立类之间**层次关系**的机制。子类能够继承父类的属性和方法，避免某些共有部分的代码被重复编写。

**例子：**

“动物”是一个很宽泛的类。它可以有“移动”和“吃”等方法，有“年龄”和“体重”等属性。

- “狗”是动物的一种，它自动拥有“年龄”和“体重”，也会“移动”和“吃”。除此之外，狗还可以有自己特有的行为，比如“摇尾巴”或“吠叫”。
- “鸟”也是动物的一种，它除了拥有动物的通用特征外，还可以有自己特有的行为，比如“下蛋”或“飞翔”。

在这里，“动物”就是父类，“狗”和“鸟”就是子类，它们“继承”了父类的特性。

**优点：**

- **代码重用**：子类可以直接使用父类的代码，避免了重复编写。
- **扩展性**：可以在不修改父类的情况下，通过创建子类来添加新的功能。
- **建立层次结构**：使代码结构更清晰，更符合现实世界的分类逻辑。

#### 3. 多态

多态的意思是“多种形态”，它指的是**同一个方法调用，在不同的对象上会有不同的行为**。

**例子：**
还是用“动物”的例子。我们向不同的动物发出一个“叫”的指令（同一个方法调用）。

- 对着一只“狗”发出“叫”的指令，它会“汪汪”。
- 对着一只“猫”发出“叫”的指令，它会“喵喵”。
- 对着一只“牛”发出“叫”的指令，它会“哞哞”。

同一个“叫”的指令，因为接收指令的动物对象不同，最终表现出的“叫声”形态也不同。你（调用者）不需要关心具体是哪种动物，只需要知道它会“叫”就行了。

注意⭐：必须是通过子类指针（即某一具体动物）来调用父类方法（即所有动物共有的“叫”的指令）,并且子类重写了父类的方法（即某一具体动物有自己的特有的叫声，否则使用动物父类默认的叫声）时，才体现出多态。可以理解为父类的“叫”指令，会根据不同的子类，实现出子类特定的效果。

### 三、面向过程编程和面向对象编程的区别⭐

这里以“**将大象装进冰箱**”为例，我们都知道，这个操作有三步：

1. 打开冰箱门
2. 把大象装进冰箱
3. 关闭冰箱门

#### 1. 面向过程编程

如果是以面向过程编程来实现的话，那我们需要编写的代码如下：

~~~c++
// 第1步：定义相关数据来描述冰箱和大象
struct Fridge {
    ...
};

struct Elephant {
    ...
};

// 第2步：定义打开冰箱门的函数
void openFridge(Fridge fridge) {
    ...
}

// 第3步：定义把大象装进冰箱的函数
void fill(Fridge fridge, Elephant elephant) {
    ...
}

// 第4步：定义关闭冰箱门的函数
void closeFridge(Fridge fridge) {
    ...
}

// 第5步：调用相关过程
int main() {
    Fridge fridge;
    Elephant elephant;
    openFridge(fridge);
    fill(fridge, elephant);
    closeFridge(fridge);
}
~~~

#### 2. 面向对象编程

如果是以面向过程编程来实现的话，那我们需要编写的代码如下：

~~~c++
// 第1步：定义填充物类
class Filler {
    填充物共有的属性和方法...
}

// 第2步：由填充物类派生出大象类，也称大象类继承自填充物类，即填充物类是父类，大象类是子类
class Elephant : public Filler {
    大象特有的的属性和方法...
}

//  第3步：定义容器类，在其内定义容器共有的属性和方法
class Container {
private:
    相关描述数据...

public:
    // 打开容器
    void open() {
        ...
    }
    
    // 装入填充物
    void load(Filler filler) {
        ...
    }
    
    // 关闭容器
    void close() {
        ...
    }
}

// 第4步：由容器类派生出冰箱类，也称冰箱类继承自容器类，即容器类是父类，冰箱类是子类
class Fridge : public Container {
private:
    冰箱特有的相关描述数据...

public:
    // 打开冰箱（可以重写，即为打开冰箱门特别定制打开操作；不重写的话与父类保持一致）
    void open() {
        ...
    }
    
    // 装入填充物（同理）
    void load(Filler filler) {
        ...
    }
    
    // 关闭冰箱（同理）
    void close() {
        ...
    }
    
    冰箱其他特有的方法...
}

// 第5步：由定义的类创建出实例化对象，并调用相关方法
int main() {
    // 实例化一个大象的对象作为填充物
    Filler* filler = new Elephant();
    // 实例化一个冰箱作为容器
    Container* container = new Fridge();
    
    //调用对象的方法
    container->open();
    container->load(filler);
    container->close();
    
    // 销毁实例化对象，释放他们占用的内存
    delete filler;
    delete container;
}
~~~

#### 3. 面向对象编程相对于面向过程编程的优势

就上面例子来看，面向对象编程似乎比面向过程编程更麻烦了。但这是在软件规模较小的情况下。

设想一下，倘若现在不仅要把大象装入冰箱了，还要把鸡蛋、苹果、牛奶放进冰箱，或者把大象放入箱子中，又或者把鸡蛋、苹果、牛奶放进篮子里。

如果使用**面向过程编程**的话，需要先定义鸡蛋、苹果、牛奶、箱子、篮子等事务相应数据的结构体，再编写打开箱子、关闭箱子、打开篮子、关闭篮子、冰箱/箱子/篮子装入鸡蛋/苹果/牛奶/大象的函数....这样就太麻烦了，存在大量重复的代码，**代码复用性差**。若装入步骤出错，也将意味着需要修改多个函数，**可维护性差**。若未来有更多的物品需要装入，或者有更多的容器，那么需要编写的函数数量将会成倍增长，**可扩展性差**。

而如果使用**面向对象编程**的话，则只需要从填充物类派生出鸡蛋、苹果、牛奶等，从容器类派生出篮子、箱子等。若篮子、箱子等的打开、装入、关闭等逻辑与父类一致的话，甚至不需要重写相应函数。这样，代码的复用性、可扩展性、可维护性都大大提高了。
